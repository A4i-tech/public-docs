    Universal Knowledge Ingestion Framework

    Purpose and Scope¶

    This document proposes a common, inclusive knowledge ingestion framework that can be used for digital public AI good:

    • BYOEB (ASHA WhatsApp RAG for healthcare)
    • Shiksha Copilot (lesson outcome–driven education copilot)
    • SEEDS (audio-first IVR and conferencing for visually impaired learners)

    …and is extensible to new products and modalities.

    It aligns with the Inclusive Architecture’s Ingestion Layer and its agent/tool model — “ingestion trigger → retrieval → parsing → chunking → embedding → storage → logging” — and implements those as reusable agents and tools across domains.

    Layered Architecture¶

    Design drivers¶

    1. Aligned with Inclusive Architecture

      • Ingestion is a layer serving multiple domains (education, health, accessibility) via a common set of agents & tools.
      • All products talk to ingestion through consistent APIs/events; ingestion itself is multi-tenant and domain-aware.

    2. Multi‑Modal

      • Documents & text: PDFs, DOCX, plain text, HTML, FAQs.
      • Audio: IVR recordings, TTS outputs, podcasts/YouTube, etc.
      • Future: images, structured data (CSV, logs), etc.

    3. Multi‑Model & Multi‑Store

      • Plug-and-play LLM providers (Azure OpenAI, OpenAI, OpenRouter, local models via Ollama).
      • OpenRouter as a default unified gateway: Access 100+ models (Claude, Gemini, Llama, Mistral) through a single API, enabling cost optimization and model experimentation without provider lock-in.
      • Pluggable vector stores (Azure Vector Search, Qdrant, Chroma, pgvector…).
      • Other stores: blob/file storage, relational/NoSQL metadata, graph DB (Neo4j), and search indexes.

    4. Domain-Aware but Reusable

      • Shiksha's LO extraction and curriculum KG should be profiles on top of generic steps, not one-off pipelines.
      • BYOEB's KB ingestion should be the default text ingestion profile.
      • SEEDS' audio/transcription/knowledge items flow should be the default audio profile.

    5. Inclusive & Operational

      • Support low-resource environments (local vector stores, offline-first when possible).
      • Strong observability, versioning, audit, and safety/quality checks.

    Conceptual Model¶

    We define three layers inside the ingestion system (itself mapped to the “Ingestion Layer” in the Inclusive Architecture):

    1. Source & Trigger Layer: Where content comes from and why we ingest it.
    2. Processing Pipeline Layer (Agents & Tools): How content is parsed, cleaned, enriched, and encoded as knowledge.
    3. Storage & Registration Layer: Where the knowledge representations live and how they are registered for use by RAG/agents.

    Canonical "Knowledge Item"¶

    All pipelines ultimately produce a canonical KnowledgeItem (concept, not implementation-bound) with:

    Field Description
    id UUID
    tenant_id Tenant/partner/org context
    domain_profile education_shiksha / health_byoeb / accessibility_seeds / etc.
    modality text, document, audio, mixed
    source_uri Blob URI / URL / IVR recording ID
    metadata Board/grade/subject/chapter, language, tags, timestamps
    structured_views LO trees, KG nodes, transcript segments
    embeddings Vector store pointers (no raw vectors in Postgres)
    lineage Pipeline version, model version, ingestion time

    BYOEB’s TextNode + vector metadata, Shiksha Copilot's StepResult and SEEDS’ KnowledgeItem schema can be mapped into this conceptual model.

    High-Level Ingestion Flow¶

    We define a generic pipeline that all domain profiles reuse:

    1. Ingestion Trigger
    2. Data Retrieval
    3. Parsing & Preprocessing
    4. Chunking & Metadata Enrichment
    5. Embedding & Indexing
    6. Storage & Caching
    7. Event Logging & Completion

    We implement each step as one or more agents (reasoning/LLM heavy) and tools (IO or deterministic work).

    Core Implementation¶

    Source And Trigger Layer¶

    Triggers: - API calls (e.g., /vector/index in BYOEB) - Webhook/events: “file uploaded to blob”, “new textbook version published”, “audio recording completed from IVR” - Scheduled jobs: “sync YouTube playlist weekly”, “re-ingest content when embedding model changes”, - Manual CLI/ops triggers

    Source Connectors: A standard source connector would implement the interface 

    interface SourceConnector {
  listResources(filter): Promise<SourceResource[]>;
  fetch(resourceId): Promise<RawArtifact>;  // bytes + metadata
}

    Examples: - AzureBlobConnector — already exists in BYOEB. - HttpUrlConnector — general web pages / PDFs. - ExternalMediaConnector — YouTube, podcasts, RSS, S3/GCS, etc., as in SEEDS’s ExternalSource model and audioDownloaderService. - IVRRecordingConnector — SEEDS’ IVR recordings; uses IVR backend IDs.

    Processing Pipeline Layer (Agents & Tools)¶

    We define a Pipeline Orchestrator that runs a graph of steps, where each step is an IngestionAgent or/and IngestionTool with explicit input/output contracts. This reuses the agent & tool library philosophy from the Inclusive Architecture.

    Common Agents and Tool types 1. Parser Agents: - DocumentParserAgent: (PDF, DOCX → markdown/structured JSON) - Implements Shiksha's TOC discovery, splitting, and LLM/vision-based extraction as sub-steps. - Supports alternative engines (MinerU, OCR). - TextParserAgent: (plain text, HTML, FAQs → normalized text). Generalizes BYOEB's LlamaIndex text parsing. - AudioParserAgent (audio → transcription + timestamps). Wraps SEEDS' TranscriptionService (Azure STT).

    1. Cleaning & Normalization Agents:
    2. MarkdownCleaningAgent — Shiksha’s TextCleaningStep (LLM-powered cleaning) generalized to all markdown sources.

    3. TranscriptionPostProcessorAgent — punctuation, diarization, noise trimming, language detection for audio transcripts.

    4. Enrichment & Domain Agents

    5. LearningOutcomeExtractionAgent — Shiksha LO extraction steps (chapter-level, subtopic-level) exposed as a reusable agent that works on any curriculum-like content.
    6. KnowledgeGraphBuilderAgent — uses Shiksha KG steps (entities, relationships, Neo4j export).
    7. FAQGenerationAgent — optional; generate FAQs for BYOEB or SEEDS from long docs/audio summaries.

    8. KeywordAndTaggingAgent — used in SEEDS to derive keywords/tags; can be reused elsewhere.

    9. Chunking Agents

    10. SentenceChunkingAgent — wraps BYOEB's LlamaIndex SentenceSplitter with configurable chunk_size / overlap.
    11. PageChunkingAgent — Shiksha's page-based splitting for textbooks.

    12. TimedChunkingAgent — for audio segments (based on transcript timestamps) to create semantically and temporally aligned chunks for SEEDS.

      How Time-Based Chunking Works: 1. The AudioParserAgent produces a transcript with word-level or segment-level timestamps (e.g., from Azure STT's word_timings). 2. TimedChunkingAgent receives this timestamped transcript and applies configurable rules: - Fixed duration windows: Split every N seconds (e.g., 30s chunks) with optional overlap. - Silence/pause detection: Use natural pauses (>500ms) as chunk boundaries. - Semantic + temporal hybrid: Combine sentence boundaries with time limits — prefer splitting at sentence ends, but force-split if duration exceeds max (e.g., 60s). 3. Each resulting chunk carries metadata: { "start_time": 45.2, "end_time": 72.8, "duration": 27.6 }. 4. These timestamps enable audio seek during playback in SEEDS IVR/teacher experiences — users can jump to the relevant audio segment for any retrieved chunk.

    13. Embedding & Indexing Agents

    14. EmbeddingAgent — wraps any embedding model; configuration chooses:
      • Provider: Azure OpenAI (current), OpenAI API, local model, etc.
      • Model: text-embedding-3-large, others; embedding dimension recorded in metadata.
    15. VectorIndexingAgent — writes to pluggable vector stores.
    16. GraphIndexingAgent — writes to Neo4j (for curriculum KG).

    17. KeywordIndexingAgent — writes full-text indexes (e.g., Mongo text indexes used in SEEDS).

    18. Governance & Logging Agents

      • IngestionLoggerAgent — logs events, durations, outcomes (success/fail).
      • DataLineageAgent — persistent record with full traceability of the chain including sources, pipeline and versions, LLMs, vector store, etc. This enables reproducability, impact analysis, audit compliance and debugging.
      • QualityVerifierAgent — sample-based QA (e.g., compare extracted text with original pages; check LO consistency).

    Pipeline Orchestration¶

    Static Orchestration with LLM-Powered Steps:

    The ingestion framework uses declarative, static pipelines — NOT LLM-driven orchestration. This is a deliberate design choice:

    Aspect Our Approach Alternative (LLM Orchestration)
    Step sequencing Defined in YAML, runs sequentially LLM decides next step dynamically
    LLM usage Within individual steps (parsing, cleaning, extraction) Also for routing/decision-making
    Predictability High — same input produces same flow Variable — LLM may choose different paths
    Cost Lower — LLM calls only where needed Higher — additional reasoning calls
    Debuggability Easy — trace through known steps Harder — must understand LLM's decisions
    Auditability Full — pipeline version = exact behavior Partial — depends on LLM reasoning logs

    Why static orchestration? - Ingestion is a batch/background process where predictability and reproducibility matter more than flexibility. - Each step already uses LLMs where valuable (vision-based parsing, text cleaning, LO extraction). - Pipeline versioning ensures identical behavior across runs — critical for compliance and debugging. - Cost-efficient: No "reasoning overhead" for routing decisions.

    Example: YAML, JSON of a pipeline described as declarative configs 

    pipeline_id: shiksha_textbook_v1
domain_profile: education_shiksha
input_modality: document
steps:
  - agent: DocumentParserAgent
    config: { strategy: "llm_vision" }
  - agent: MarkdownCleaningAgent
  - agent: LearningOutcomeExtractionAgent
  - agent: SentenceChunkingAgent
    config: { chunk_size: 1024, overlap: 100 }
  - agent: EmbeddingAgent
    config: { model: "text-embedding-3-large", provider: "azure" }
  - agent: VectorIndexingAgent
    config: { store: "qdrant" }
  - agent: KnowledgeGraphBuilderAgent
    config: { graph_store: "neo4j" }

    BYOEB’s current KBService.upload() orchestration (download → parse → similarity check → add to vector store) becomes one such pipeline (health_kb_v1), and its BaseVectorStore abstraction can be moved into the shared library.

    SEEDS’ “audio → transcription → keywords → embedding → KnowledgeItem” flow becomes accessibility_audio_v1.

    Storage & Registration Layer¶

    Storage Layer 1. Object Storage 

    interface ObjectStore {
  put(bytes, key, metadata?): Promise<string>;  // returns URI
  get(key): Promise<RawArtifact>;
}
    - Implementation: Azure Blob (existing in BYOEB & SEEDS). - Used for PDF chapters, processed markdown, audio files, etc.

    1. Vector Store 
      interface VectorStore {
  upsert(vectors: VectorRecord[]): Promise<void>;
  search(query_vector, k, filters?): Promise<VectorRecord[]>;
  count(): Promise<number>;
}
    2. Backends:
      • Azure Vector Search (BYOEB).
      • Qdrant (Shiksha).
      • Chroma / pgvector for local/offline scenarios.

    3. Reuse BYOEB's BaseVectorStore design and vector upsert/similarity logic (including 0.95 similarity de-duplication).

    4. Graph Store 

      interface GraphStore {
  upsertNodes(nodes: Node[]);
  upsertEdges(edges: Edge[]);
}

    5. Neo4j backing for Shiksha’s KG export.

    6. Metadata Store

    7. Prefer a common metadata/entity store (e.g., Mongo/Postgres) that holds KnowledgeItem docs referencing object/vector/graph layers.

    8. SEEDS’ KnowledgeItem model is a good starting point, extended slightly to match the canonical KnowledgeItem spec.

    9. Registration and Discovery

    10. After ingestion, each pipeline registers a KnowledgeItem (or batch) into a Knowledge Registry:
    11. Saves core attributes (domain_profile, modality, metadata).
    12. Records pointers to:
      • Blob URIs for raw/processed files.
      • Vector store IDs / collections.
      • Graph DB node IDs.
    13. Agents (e.g., RAG agents in Experience Layer) ask a KnowledgeBaseAdapter tool for:
      • get_items(domain_profile, filters)
      • semantic_search(query, profile, k, filters)
      • get_by_id(id)

    This is the single consistent entrypoint for BYOEB chat, Shiksha copilot, and SEEDS IVR/teacher experiences to reuse the same knowledge.

    Domain Profiles¶

    Domain profiles use YAML schema with multi tenant isolation & secrets

    1. YAML: Public profiles live in git and describe what to do, not how to authenticate.

    2. Secretful Config (private): A separate, non-public config (env or private YAML) maps these logical refs to real credentials.

    Multi-Tenant Support¶

    Every pipeline run is created with a tenant. The runner uses: - tenant.slug plus template strings in YAML to produce: - Vector index name / collection. - Graph DB database/namespace. - Postgres tables always carry tenant_id, so queries are scoped.

    That gives: - Logical separation in all stores. - Ability to run a single profile across many tenants.

    Tenant–Pipeline Relationship:

    A single tenant can use multiple pipelines simultaneously. For example, a state education board tenant (kseeb_grade6) might use: - shiksha_textbook_v1 for textbook ingestion - shiksha_audio_v1 for supplementary audio content - generic_faq_v1 for FAQ documents

    The pipeline_runs table links each run to both tenant_id and pipeline_version_id, so: - Tenants can mix-and-match pipelines as needed. - Each pipeline can be versioned independently. - Retrieval queries can filter by domain_profile or span across all knowledge for that tenant.

    Pipeline YAML configs are versioned in git (e.g., profiles/shiksha_textbook_v1.yaml). The pipeline_versions table's yaml_profile_name field references this file. When a pipeline evolves, we create a new version (v1.1, v2.0) rather than modifying the existing YAML, ensuring reproducibility and auditability.

    How This Hooks Back into the Orchestration Layer¶

    From the Orchestrator's point of view, Ingestion is just another tool:

    • Tool: ingest_content
      • Inputs: { tenant_slug, pipeline_name, pipeline_version? , source_uri, extra_params }
      • Output: { pipeline_run_id, knowledge_item_ids[] }
    • Tool: search_knowledge
      • Inputs: { tenant_slug, domain_profile, query, filters }
      • Uses Postgres + configured vector store to:
        • pick the right indexes (vector_store_space)
        • call vector search
        • hydrate results from chunks and knowledge_items.

    This gives one Knowledge Base interface for all three systems & any new ones.

    Backward Compatibility & Versioning Strategy¶

    Register Current Workflows as v0.x Pipelines¶

    BYOEB - Treat the current KBService.upload() + Azure Blob download as pipeline byoeb_kb version 0.1. - The FastAPI endpoint /vector/index continues to call kb_upload(). Internally, we: - Call the new PipelineRunner with pipeline_version='0.1'. - Use the BYOEBKBUploadStep adapter which mostly just wraps existing behaviour. - Pipe logs into Postgres pipeline_runs / pipeline_step_runs as well as existing logs.

    Shiksha Copilot - The current pipeline_runner.py config is effectively pipeline shiksha_textbook version 0.1. - We wrap its steps to conform to the new Step protocol and run it under the Ingestion Service for new ingestions while still allowing CLI-style runs.

    SEEDS No live code yet, so first implementation can start directly as seeds_audio version 1.0.

    Introducing v1.x Pipelines¶

    For each domain we can then evolve: byoeb_kb v1.0: Maybe adds translation agent, or uses a different parser.In new framework:

    Connectors - AzureBlobConnector using existing async blob client.

    Steps - BYOEBListFilesStep (optional) → lists & filters FileMetadata. - BYOEBDownloadStep wrapping _abulk_download_files. - BYOEBParseChunksStep wrapping text_parser.get_chunks_from_collection(...). - BYOEBUpsertStep wrapping _gather_similar_chunks + _add_nodes_to_vector_store.

    shiksha_textbook v1.0: Moves to Postgres-backed registry; uses improved chunking for better RAG. In new framework:

    Each of Steps 3–8 (text extraction, cleaning, LO extraction, subtopic cleaning, subtopic-wise LO extraction, create index) becomes a Step in the Ingestion Service.

    The CreateIndexStep additionally: - Writes knowledge_items row for the chapter, with metadata {board, grade, subject, chapter_number, medium}. - Writes chunks rows for each page/subtopic chunk with Qdrant pointers.

    Optional KG steps become a separate pipeline (shiksha_kg_v1) that reads from the same cleaned markdown and writes to the graph store.

    The Shiksha “experience layer” agents then call the KnowledgeBase tool using board/grade/subject or a combination of all 3 filters that are now available across all stores and Postgres.

    seeds_audio v1.0: First implementation. Because each knowledge_item row carries pipeline_version_id, retrieval logic can behave differently if necessary (e.g., interpret metadata shapes slightly differently). In new framework:

    • ExternalSource + audioDownloaderService → SourceConnectorimplementations.
    • Transcription and keyword extraction steps → agents in the pipeline.
    • KnowledgeItem DB model becomes a tenant-specific view of the generic knowledge_items table (we can keep a Mongo version or migrate to Postgres gradually).

    The canonical knowledge_items row for an audio piece looks like: 

    {
  "domain_profile": "accessibility_seeds",
  "modality": "audio",
  "source_uri": "blob://seeds/audio/123.mp3",
  "metadata": {
    "language": "kn-IN",
    "theme": "math lesson",
    "type": "story",
    "duration": 300,
    "tags": ["grade6", "fractions"],
    "transcript_available": true
  }
}

    And chunk rows correspond to transcript segments, with timestamps in metadata.

    generic_document v1.0: A starter profile for new applications onboarding onto the ingestion framework.

    This profile provides sensible defaults that work for most text/document use cases without requiring domain-specific customization:

    pipeline_id: generic_document_v1
domain_profile: generic
input_modality: document
description: "Default pipeline for new applications - handles PDFs, DOCX, text files"
steps:
  - agent: DocumentParserAgent
    config: 
      strategy: "auto"  # auto-detect: use OCR for scanned, direct extraction for digital
      fallback_to_ocr: true
  - agent: MarkdownCleaningAgent
    config:
      preserve_tables: true
      preserve_lists: true
  - agent: SentenceChunkingAgent
    config: 
      chunk_size: 512
      overlap: 50
  - agent: EmbeddingAgent
    config: 
      model: "text-embedding-3-small"  # cost-effective default
      provider: "azure"
  - agent: VectorIndexingAgent
    config: 
      store: "qdrant"  # or "${tenant.default_vector_store}"

    New applications can: 1. Start with generic_document_v1 to get basic ingestion working immediately. 2. Fork and customize — copy the YAML, add domain-specific agents (e.g., FAQGenerationAgent), adjust chunking parameters. 3. Graduate to a named profile — once stable, register as myapp_kb_v1 with domain_profile myapp.

    This lowers the barrier to onboarding while maintaining the full power of the framework for teams ready to customize.

    Low Level Design¶

    Postgres Metadata & Registry Design¶

    We’ll put the “brain” of ingestion metadata in Postgres and keep vectors in the vector stores.

    Core Tables¶

    tenants 

    CREATE TABLE tenants (
  id              UUID PRIMARY KEY,
  slug            TEXT UNIQUE NOT NULL,  -- e.g. 'asha_raj', 'kseeb_grade6'
  display_name    TEXT NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

    pipelines – logical pipelines (e.g. byoeb_kb, shiksha_textbook, seeds_audio) 

    CREATE TABLE pipelines (
  id              UUID PRIMARY KEY,
  name            TEXT NOT NULL,           -- 'byoeb_kb', 'shiksha_textbook'
  domain_profile  TEXT NOT NULL,           -- 'health_byoeb', 'education_shiksha', ...
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

    pipeline_versions 

    CREATE TABLE pipeline_versions (
  id                UUID PRIMARY KEY,
  pipeline_id       UUID REFERENCES pipelines(id),
  version           TEXT NOT NULL,          -- '0.1', '1.0', '1.1'
  yaml_profile_name TEXT NOT NULL,          -- name of the YAML in git
  is_default        BOOLEAN NOT NULL DEFAULT false,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

    pipeline_runs – like Shiksha’s pipeline runner, but shared and tenant-aware 

    CREATE TABLE pipeline_runs (
  id                UUID PRIMARY KEY,
  tenant_id         UUID REFERENCES tenants(id),
  pipeline_version_id UUID REFERENCES pipeline_versions(id),
  source_uri        TEXT NOT NULL,  -- blob path, http URL, etc.
  status            TEXT NOT NULL,  -- 'queued','running','succeeded','failed'
  started_at        TIMESTAMPTZ,
  completed_at      TIMESTAMPTZ,
  error_message     TEXT
);

    pipeline_step_runs – thin wrapper around Shiksha’s StepResult 

    CREATE TABLE pipeline_step_runs (
  id                UUID PRIMARY KEY,
  pipeline_run_id   UUID REFERENCES pipeline_runs(id),
  step_name         TEXT NOT NULL,          -- 'text_extraction_llm', 'kb_upload', ...
  status            TEXT NOT NULL,          -- matches StepStatus
  output_paths      JSONB,                  -- type->path (for file-based outputs)
  metadata          JSONB,
  error_message     TEXT,
  started_at        TIMESTAMPTZ,
  completed_at      TIMESTAMPTZ
);

    knowledge_items – canonical registry entry 

    CREATE TABLE knowledge_items (
  id                  UUID PRIMARY KEY,
  tenant_id           UUID REFERENCES tenants(id),
  domain_profile      TEXT NOT NULL,
  modality            TEXT NOT NULL,            -- 'text','document','audio','mixed'
  source_uri          TEXT NOT NULL,            -- blob/URL/IVR ID
  pipeline_version_id UUID REFERENCES pipeline_versions(id),
  title               TEXT,
  description         TEXT,
  metadata            JSONB,                    -- board, grade, subject, tags...
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

    chunks – registry for text/semantic units (BYOEB’s Chunk / LlamaIndex TextNode) 

    CREATE TABLE chunks (
  id                  TEXT PRIMARY KEY,         -- reuse chunk_id/node_id
  knowledge_item_id   UUID REFERENCES knowledge_items(id),
  tenant_id           UUID REFERENCES tenants(id),
  text                TEXT NOT NULL,
  metadata            JSONB,
  vector_store_kind   TEXT NOT NULL,            -- 'azure_vector_search','qdrant',...
  vector_store_space  TEXT NOT NULL,            -- index/collection/namespace name
  vector_store_ref_id TEXT NOT NULL,            -- id/primary key inside store
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

    We don’t store the vector in Postgres; we just store the pointer to wherever that vector lives.

    For SEEDS, a KnowledgeItem will additionally carry transcription bits in metadata or a sibling audio_segments table later.

    Pipeline & Step Abstractions¶

    Step Interface

    We unify BYOEB’s monolithic KBService.upload() and Shiksha’s multi-step pipeline by introducing a generic Step that always returns a Shiksha-style StepResult. 

    class Step(Protocol):
    name: str

    async def run(self, ctx: IngestionContext) -> StepResult:
        ...

    Where: 

    @dataclass
class StepResult:   # from Shiksha, reused
    status: StepStatus
    output_paths: Dict[str, str] | None = None
    error: Exception | None = None
    metadata: Dict[str, Any] | None = None

    IngestionContext is a mutable object holding: - tenant, pipeline_version, pipeline_run_id - references to raw artifacts - intermediate outputs (e.g., extracted markdown path) - a Postgres connection and store registries.

    Step Types

    We’ll create adapters so existing code can be treated as Steps.

    -BYOEB KB Upload Step

    Wrap the existing KBService.upload() to behave like a pipeline step.

    class BYOEBKBUploadStep(Step):
    name = "byoeb_kb_upload"

    async def run(self, ctx: IngestionContext) -> StepResult:
        kb_service = KBService(
            vector_store=ctx.stores.vector_store,
            media_storage=ctx.stores.media_storage,
            llm_client=ctx.models.llm,
            upsert_t=ctx.config.upsert_threshold,
        )
        count = await kb_service.upload(ctx.inputs.files_metadata)
        return StepResult(
            status=StepStatus.SUCCESS,
            metadata={"chunks_ingested": count}
        )

    The orchestrator then records a pipeline_step_runs row with this metadata.

    -Shiksha Steps

    Each existing Shiksha Step (TOC, text extraction, text cleaning, LO extraction, index creation, KG) already conforms conceptually: they write outputs to files and can populate StepResult.output_paths.

    We just ensure: - They accept an IngestionContext. - They fill StepResult and allow the orchestrator to record them.

    -SEEDS Steps

    For SEEDS, we’ll add steps like: - AudioDownloadStep (pull from blob/external URL) - TranscriptionStep - KeywordExtractionStep - AudioChunkingStep - EmbeddingIndexingStep - KnowledgeItemRegistrationStep

    All of them return StepResult, write rows to Postgres, and index vectors appropriately.

    Pipeline Orchestrator¶

    The Ingestion Service runs pipeline versions from YAML configs using these steps. We keep it conceptually simple: 

    class PipelineRunner:
    async def run(self, tenant, pipeline_version, source_uri, params):
        run_id = insert_pipeline_run(...)
        ctx = IngestionContext(...)

        for step in pipeline_version.steps:
            res = await step.run(ctx)
            insert_pipeline_step_run(run_id, step.name, res)
            if res.status is StepStatus.FAILURE:
                mark_run_failed(run_id, res.error)
                break

        # On success, create KnowledgeItem rows, Chunk rows, etc.

    This is where backward compatibility lives – old workflows become specific pipeline versions.

    Future Extensions¶

    Image & Multimodal Ingestion - Vision OCR for handwritten content and diagrams - Object/diagram extraction and description generation - Multimodal embeddings (e.g., CLIP-style) for image+text retrieval

    Multi-Lingual Content Handling - Language detection at ingestion time (auto-tag metadata.language) - Language-specific chunking rules (e.g., different tokenization for Hindi/Kannada vs English) - Cross-lingual embeddings or language-specific embedding models - Translation agents for creating parallel content in multiple languages - Script handling (Devanagari, Kannada script) for audio transcription post-processing

    MCP-Based Ingestion Triggers - Expose ingestion as an MCP (Model Context Protocol) server, allowing LLM agents to directly invoke ingest_content and search_knowledge tools - Enable agentic workflows where an AI assistant can autonomously decide to ingest new content (e.g., "I found a relevant PDF, let me add it to the knowledge base") - Support MCP resource discovery — agents can list available pipelines and their capabilities

    Cross-Domain Knowledge Graph - Unified graph that connects health, education, and accessibility knowledge - Enable cross-domain queries (e.g., "health topics relevant to grade 6 science curriculum") - Shared entity resolution across domains

    Pipeline Selector Agent - For highly dynamic content (e.g., mixed-format uploads where the optimal pipeline varies), introduce an optional PipelineSelectorAgent - Uses an LLM to analyze incoming content and select the most appropriate pipeline (e.g., "this looks like a scanned textbook → use shiksha_textbook_v1" vs "this is a plain FAQ document → use generic_document_v1") - The selected pipeline still runs statically — LLM only assists in the initial routing decision - Useful for self-service upload portals where users don't specify content type
